# Root Issue Template 項目定義

`assets/root-issue.template.md` の各項目に、何をどう書くかを定義する。
このガイドは root issue 本文作成時の基準として使う。

## 共通ルール

1. 主語は「この root で何を達成するか」を明確にする。
2. 抽象語だけで終わらせない。確認可能な条件を入れる。
3. レビューで意思決定できる情報を必ず残す。

## 項目別定義

### `sprint_days`

- 何を書くか: 当該 root のレビュー完了までに使う営業日数。
- 最低条件: `1` 以上 `7` 以下の整数。
- 良い例: `sprint_days: 3`
- 悪い例: `sprint_days: 2-3`, `sprint_days: 1週間`

### `スプリントゴール`

- 何を書くか: この root で完了させる成果を 1-2 行で書く。
- 最低条件: 「対象」と「完了状態」が入っている。
- 良い例: `Porters API の認証と候補者取得を実装し、ステージング環境で日次実行を成功させる。`
- 悪い例: `実装を進める。`

### `ビジネス価値`

- 何を書くか: ビジネス側がレビューで確認できる価値。
- 最低条件: 価値の受け手と確認観点が明示されている。
- 良い例: `スカウト送付対象の抽出根拠をレビューで確認でき、手動選定工数を比較できる。`
- 悪い例: `価値がある。`

### `レビュー手順`

- 何を書くか: レビュー時に誰が何をどう確認するか。
- 最低条件: 2 手順以上。確認対象と判定観点が入っている。
- 良い例: `1. Biz 担当が抽出対象 30 件の妥当性を確認する。`
- 良い例: `2. Eng 担当が API 失敗率と再試行結果を確認し、次スプリントの優先度を決める。`
- 悪い例: `レビューする。`

### `完了条件`

- 何を書くか: root をクローズしてよい判定条件。
- 最低条件: 検証可能な条件を複数行で書く。
- 良い例: `子 issue が全て Closed で、Blocker ラベルの未解決件数が 0 件である。`
- 良い例: `スプリントレビュー議事録に Go/No-Go 判定と次アクションが記録されている。`
- 悪い例: `だいたい終わっている。`

### `背景と目的`

- 何を書くか: なぜこの root が必要か。
- 最低条件: 課題と目的が 1 つ以上ずつ書かれている。
- 良い例: `手動送付では日次 2000 件を処理できないため、自動抽出と送付判定の基盤を先に成立させる。`
- 悪い例: `必要だから。`

### `スコープ（やること）`

- 何を書くか: この root で実施する範囲。
- 最低条件: 箇条書きで 2 項目以上。
- 良い例: `- Porters API 認証の実装`
- 良い例: `- 候補者取得バッチの実行基盤構築`
- 悪い例: `- いろいろ対応`

### `非スコープ（やらないこと）`

- 何を書くか: 今回扱わない範囲。
- 最低条件: 今回除外する内容を 1 項目以上。
- 良い例: `- 本番運用の 24 時間監視体制の整備`
- 悪い例: `- なし`

### `制約（期限、技術、運用、コスト）`

- 何を書くか: 期限・技術・運用・コストの制約条件。
- 最低条件: 少なくとも 2 種類以上の制約を列挙する。
- 良い例: `- 06:00 JST までに日次処理を完了する。`
- 良い例: `- Porters API の Read/Write 制限を超過しない。`
- 悪い例: `- 制約に注意する。`

### `リスクまたは未確定事項`

- 何を書くか: 未確定要素、前提崩れ時のリスク。
- 最低条件: リスク内容と影響範囲がわかる。
- 良い例: `- Porters API 項目定義変更時に抽出条件が無効化するリスクがある。`
- 悪い例: `- リスクあり`

### `依存する Root Issue`（任意）

- 何を書くか: 先に完了が必要な root と依存理由。
- 最低条件: root_id か issue 番号が識別できる。
- 良い例: `- root-01-foundation (DB スキーマ確定待ち)`

### `ブロックする Root Issue`（任意）

- 何を書くか: この root の完了待ちになる後続 root。
- 最低条件: root_id か issue 番号が識別できる。
- 良い例: `- root-03-validation (評価データ抽出処理の実装待ち)`
